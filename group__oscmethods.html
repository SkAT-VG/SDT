Detailed Description

Implementation of OSC methods for SDT

Ordinary classes

Most SDT classes can be controlled via OSC in the same way. Their address space is defined by the following pattern:

/<class>/<method>

Setter methods

Each class has a setter method for each of their attributes. The method name for setter methods is the attribute name. The arguments for the setter methods are:

/<class>/<attribute> <key> <value>

Please note that, for a SDT object to be controllable via OSC, it has to be registered with a key. See the SDT_register<class>() method of the class of interest for further details.

JSON methods

Each class also has methods for JSON serialization and deserialization.

  • log logs the JSON description of an object to the configured info-level stream
  • save saves the JSON description of an object to a file
  • load loads the JSON description of an object from a file
  • loads loads the JSON description of an object from a string

The first argument is always the key of the object of interest. The methods save and load also require a filepath as a second argument. The method loads interprets any following arguments as a JSON payload.

/<class>/log <key>
/<class>/save <key> <filepath>
/<class>/load <key> <filepath>
/<class>/loads <key> <json>...

Special classes

Some classes have a special behaviour, namely:

  • resonators (SDTOSCResonators) have array-valued attributes
  • interactors (SDTOSCInteractors.h: OSC Interactors) are identified by two keys
  • projects (SDTOSCProjects) allow to handle multiple objects at once with JSON
